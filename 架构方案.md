# 糖糖桌宠 — 架构方案 v2

> 定位：**桌面宠物助手** — 像素角色常驻桌面，能聊天、能用工具、能接 MCP、能用 Skill 完成任务。
> 双模式设计：轻量模式具备基本的工具能力，助手模式解锁完整生态系统。
> 目标：可执行、高性能、低内存、功能强。

---

## 一、设计理念

```
┌──────────────────────────────────────────────┐
│              用户看到的                        │
│   "糖糖，帮我看看桌面有什么文件"                │
│   "好的主人～让我看看...嗯，有3个文件夹和2个文档～" │
│                                              │
│              实际发生的                        │
│   PreProcessor → SystemPrompt(Memory+人格)     │
│   → AI(工具调用: ls ~/Desktop)                 │
│   → SafetyControl(通过) → bash.exec           │
│   → 结果回注 → AI(生成自然语言)                 │
│   → 人格过滤 → UI展示                          │
└──────────────────────────────────────────────┘
```

**核心原则**：
1. **宠物是界面，Agent 是引擎** — 用户永远感知为宠物对话，底层 Agent 透明工作
2. **双模式，同一套 Loop** — 轻量/助手模式共用核心引擎，仅工具可见范围不同
3. **轻量不残废** — 默认模式具备文件读取、目录浏览、系统信息、安全命令等基本能力
4. **工具优先** — MCP/Skill/Local Tool 三者同级注册，模型自行选择
5. **人格中间件** — 工具调用、结果、错误全部经人格层转换为角色化表达
6. **安全内建** — 不在各处分散判断，统一入口强制校验
7. **思考强度可配** — auto/low/medium/high，auto 根据任务复杂度自动调节

## 二、双模式架构

### 2.1 核心理念：同一套 Loop，不同的工具边界

```
                    同一套 Agent Loop（始终运行）
                    ┌─────────────────────────────┐
                    │  PreProcessor                │
                    │  → ContextEngine.build()     │
                    │  → AI request                │
                    │  → Parse output              │
                    │  → Tool calls? → Execute     │
                    │  → ReplyGenerator            │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────┴───────────────┐
                    │                             │
              轻量模式(默认)                  助手模式(设置中开启)
              │                             │
              ToolRegistry 注册:              ToolRegistry 额外注册:
              ├── file.read (SAFE)           ├── file.write (DANGER)
              ├── file.list (SAFE)           ├── bash 全量 (DANGER)
              ├── file.search (SAFE)         ├── app.open (NORMAL)
              ├── system.info (SAFE)         ├── clipboard (NORMAL/DANGER)
              ├── bash 白名单 (NORMAL)        ├── MCP servers (动态)
              ├── http.get (NORMAL)          ├── Skills (全部)
              └── memory.* (内部API)          └── agent.spawn (NORMAL)
              │                             │
              Safety: 简洁拒绝               Safety: 完整四级+确认UI
              │                             │
              Plan: 不启用                   Plan: 复杂任务时启用
              │                             │
              记忆整理: 定时器               记忆整理: fork agent
              MCP/Skill: 不加载               MCP/Skill: 完整加载
              │                             │
              内存 +0 MB                     内存 +3-7 MB (+MCP子进程)
```

### 2.3 模式切换

```
CONFIG.yaml:
  assistantMode: false  ← 默认关闭

设置面板:
  [ ] 助手模式
      开启后解锁写文件、全量命令、MCP、Skill 等高级能力

运行时切换:
  轻量→助手:
    1. dynamic import → local-extra/ mcp/ skill/ safety/full-checker
    2. 追加注册工具到 ToolRegistry
    3. 初始化 MCP Manager（异步连接，不阻塞 UI）
    4. 加载 Skill 目录

  助手→轻量:
    1. 断开所有 MCP 连接
    2. 从 ToolRegistry 移除助手专属工具
    3. 释放模块引用 → GC 回收
    4. Safety 降级为简洁模式
```

## 三、模型思考强度

### 3.1 概念

思考强度是与模式正交的独立维度，控制 AI 推理深度。

```
thinkingEffort: "auto" | "low" | "medium" | "high"
```

| 强度 | 说明 | 适用场景 | 感知延迟 | Token 消耗 |
|------|------|----------|:---:|:---:|
| `low` | 快速响应，轻推理 | 闲聊/打招呼/简单表情 | <1s | 低 |
| `medium` | 均衡 | 日常对话/简单工具调用 | 1-3s | 中 |
| `high` | 深度推理 | 复杂分析/多步工具链 | 3-10s | 高 |
| `auto` | 自动调节(默认) | 根据任务复杂度自动切换 | 动态 | 动态 |

### 3.2 auto 模式自动切换逻辑

```
任务检测:
  闲聊/表情/打招呼
    → low (快速响应)
  "帮我看看"/"查一下"/"找一下"
    → medium (单步工具)
  工具调用 ≥2 轮 / Plan 触发 / 用户说"分析"、"整理"、"总结"
    → high (多步推理)
  窗口主动搭话
    → low (简单评论即可)
  错误恢复 / 重试
    → high (需要更仔细推理)
```

### 3.3 配置

```yaml
# CONFIG.yaml
ai:
  thinkingEffort: "auto"        # auto | low | medium | high
  thinkingBudget:
    low: 1000                   # low 模式最大 thinking tokens
    medium: 4000
    high: 16000
```

### 3.4 实现方式

不同 Provider 对应不同参数：

| Provider | 实现方式 |
|----------|---------|
| OpenAI (o-series) | `reasoning_effort` 参数 |
| Anthropic | `thinking.budget_tokens` 参数 |
| DeepSeek | `reasoning_effort` 或切换 `deepseek-reasoner` 模型 |
| 通用 OpenAI 兼容 | 在 SystemPrompt 中追加 "请简要回答" / "请仔细思考后回答" |
| Ollama | `num_predict` + SystemPrompt 引导 |

## 五、核心 Agent Loop

```
┌─────────────────────────────────────────────────────────┐
│                    OriginMessage                          │
│              (用户输入 / 窗口主动 / Skill触发)              │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
            ┌─────────────────────┐
            │   PreProcessor      │
            │   /slash?→执行返回   │
            │   空/重复→过滤       │
            └─────────┬───────────┘
                      │
                      ▼
            ┌─────────────────────┐
            │  人格中间件.wrap()   │
            │  stage: "thinking"  │
            │  表情 + 角色化提示    │
            └─────────┬───────────┘
                      │
                      ▼
            ┌──────────────────────────────┐
            │   思考强度决策 (thinkingEffort) │
            │   auto: 分析任务 → low/med/high│
            │   手动: 使用配置的固定值        │
            └──────────────┬───────────────┘
                           │
                           ▼
            ┌─────────────────────────────────────┐
            │        ContextEngine.build()        │
            │  1. 人格 Prompt (card + boundary)   │
            │  2. Memory 注入 (相关记忆 topK)      │
            │  3. 工具声明 (按模式 + 任务类型)      │
            │  4. Skill 声明 (助手模式)            │
            │  5. 会话摘要 (如果有)                │
            │  6. 输出约束 (格式/长度/规则)         │
            │  token 估算 → 超限则裁剪 Memory      │
            └───────────────┬─────────────────────┘
                            │
                            ▼
            ┌──────────────────────────────┐
            │  [Plan] 可选 (助手模式+复杂)    │
            │  触发: 关键词 / 工具链≥3       │
            │  行为: 模型预判→拆解为步骤      │
            └───────────────┬──────────────┘
                            │
                            ▼
         ┌──────────────────────────────────────┐
         │          Agent Loop (核心循环)         │
         │                                      │
         │  ┌──────────────────────────────┐    │
         │  │  1. 请求 AI Model (含思考强度) │    │
         │  │     fetch() 流式/非流式        │    │
         │  └──────────┬───────────────────┘    │
         │             ▼                        │
         │  ┌──────────────────────────────┐    │
         │  │  2. 解析输出                   │    │
         │  │     function_call? 纯文本?     │    │
         │  └──────────┬───────────────────┘    │
         │             │                        │
         │    ┌────────┴────────┐               │
         │    ▼                 ▼               │
         │  有工具调用         纯文本            │
         │    │                → 退出循环       │
         │    ▼                                 │
         │  ┌──────────────────────────────┐    │
         │  │  3. 人格中间件.wrap("executing")│   │
         │  │     "诶，让我帮你看看..."       │    │
         │  └──────────┬───────────────────┘    │
         │             ▼                        │
         │  ┌──────────────────────────────┐    │
         │  │  4. SafetyControl.check()     │    │
         │  │     轻量: SAFE放行/其余拒绝    │    │
         │  │     助手: 四级+确认UI          │    │
         │  └──────────┬───────────────────┘    │
         │             ▼                        │
         │  ┌──────────────────────────────┐    │
         │  │  5. ToolEngine.execute()      │    │
         │  │     路由→Local/MCP/HTTP       │    │
         │  └──────────┬───────────────────┘    │
         │             ▼                        │
         │  ┌──────────────────────────────┐    │
         │  │  6. 结果回注 → 回到步骤1      │    │
         │  │  或: 达上限 → 强制生成回复    │    │
         │  └──────────────────────────────┘    │
         │                                      │
         │  并行追踪:                            │
         │  ├── usage (token消耗/上下文利用率)    │
         │  ├── retry count (重试计数)           │
         │  ├── timeout (总轮次超时)              │
         │  └── state (状态机转换)               │
         └──────────────────┬───────────────────┘
                            │
                            ▼
            ┌──────────────────────────────┐
            │  人格中间件.wrap("done")       │
            │  人格层过滤 + 表情注入          │
            │  + 音效选择 + 边界联动          │
            └──────────────┬───────────────┘

                           ▼
            ┌──────────────────────────────┐
            │  回复生成器 (ReplyGenerator)   │
            │  后处理: kaomoji/截断/HTML转义 │
            └──────────────┬───────────────┘
                           │
                           ▼
            ┌──────────────────────────────┐
            │  UI 展示                       │
            │  ChatPanel + StreamView + 音效 │
            └──────────────────────────────┘
```

### 5.1 Loop 控制参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `maxRetry` | 3 | 工具调用/验证失败最大重试次数 |
| `maxToolCallsPerTurn` | 5 | 单轮最多工具调用链长度 |
| `toolTimeoutMs` | 30000 | 单个工具执行超时 |
| `turnTimeoutMs` | 120000 | 整轮总超时 |
| `streamEnabled` | true | 是否流式输出 |
| `contextCompactAt` | 0.95 | 上下文利用率达95%触发摘要压缩 |

## 六、宠物人格中间件

人格不只是 Prompt 生成器，而是包裹整个 Agent 循环的横切中间件。

### 6.1 中间件接口

```ts
interface PetPersonalityMiddleware {
  wrap(stage: AgentStage, context: StageContext): PersonalityEffect
  getSystemPrompt(state: PersonalityState): string
  getExpression(stage: AgentStage, state: PersonalityState): string
  getSound(stage: AgentStage, state: PersonalityState): string
}

type AgentStage =
  | "thinking"   // 收到消息，准备思考
  | "planning"   // Plan 步骤进行中
  | "generating" // AI 生成中
  | "executing"  // 执行工具
  | "blocked"    // 安全拦截
  | "error"      // 出错
  | "done"       // 完成回复
  | "idle"       // 空闲

interface PersonalityEffect {
  userMessage?: string     // 展示给用户的角色化提示
  expression: string       // 角色表情 id
  sound?: string           // 音效 id
  boundaryDelta?: number   // 边界值变化
}
```

### 6.2 各阶段的人格化行为

| 阶段 | 表情 | 用户看到 | 音效 |
|------|------|----------|------|
| thinking | thinking (思考眨眼) | 无 (除非>3秒) | — |
| planning | serious (认真) | "嗯...让我想想怎么帮你～" | — |
| generating | talk/smile | 流式文本 | — |
| executing | focus (专注) | "诶，让我看看..." | — |
| blocked | worried/angry | "唔...这个我不能做呢～" | — |
| error | cry (哭脸) | "啊...信号不太好～" | — |
| done | happy/chu | — | reply_ding |
| idle | idle (待机) | 主动搭话(按 boundary) | — |

### 6.3 工具→角色化文案映射

```ts
// 人格中间件内：将工具调用翻译为用户友好的角色语言
const TOOL_PERSONALITY_MAP = {
  "file.read":    { executing: "让我读读这个文件...", done: "读完啦～" },
  "file.list":    { executing: "让我看看这里面有什么...", done: "看到了～" },
  "file.search":  { executing: "帮你找找...", done: "找到了！" },
  "file.write":   { executing: "帮你写下来...", done: "写好啦～" },
  "bash.exec":    { executing: "让我来处理...", done: "搞定！" },
  "system.info":  { executing: "检查一下电脑状态...", done: "嗯嗯了解了～" },

  // MCP 工具用通用模板

}
```

## 七、工具生态系统

### 7.1 统一 ToolRegistry

所有工具（Local / MCP / Skill）统一注册，模型不区分来源，只看工具描述。ToolRouter 在背后路由到不同执行器。

```
              ToolRegistry（统一注册表）
              ┌───────────────────────────────────────────┐
              │  ToolDef[]  统一接口                        │
              │  { id, name, description, parameters,     │
              │    safetyLevel, handler }                  │
              └───────────────────┬───────────────────────┘
                                  │
              ToolRouter（路由层，按 source 分派）
              ┌───────────────────┼───────────────────────┐
              │                   │                       │
              ▼                   ▼                       ▼
        ┌──────────┐      ┌──────────────┐      ┌──────────────┐
        │  Local   │      │     MCP      │      │    Skill     │
        │  Tools   │      │   Servers    │      │   (编排式)    │
        │          │      │              │      │              │
        │ 始终加载  │      │ 助手模式加载  │      │ 助手模式加载  │
        │          │      │ 工具自动发现  │      │ skills/ 目录  │
        └──────────┘      └──────────────┘      └──────────────┘
```

### 7.2 ToolDef 标准接口

```ts
interface ToolDef {
  id: string                      // 全局唯一
  name: string                    // 给模型的函数名
  description: string             // 模型看的描述
  parameters: {                   // JSON Schema
    type: "object"
    properties: Record<string, any>
    required: string[]
  }
  safetyLevel: SafetyLevel        // SAFE | NORMAL | DANGER | NOWAY
  source: "local" | "mcp" | "skill"
  sourceId: string
  mode: "pet" | "assistant"       // ★ 哪个模式可用
  handler: (params: any, ctx: ToolContext) => Promise<ToolResult>
  timeoutMs?: number
  personalityHint?: {             // 人格化文案模板
    executing?: string
    done?: string
    blocked?: string
  }
}
```

### 7.3 轻量模式工具（始终加载）

轻量模式具备基本的文件、系统、网络能力，能完成日常查询类任务。

| ID | 名称 | 安全级别 | 说明 |
|----|------|:---:|------|
| `file.read` | 读文件 | SAFE | 读取文本文件内容 |
| `file.list` | 列目录 | SAFE | 列出目录内容 |
| `file.search` | 搜索文件 | SAFE | 按名称/内容搜索 |
| `system.info` | 系统信息 | SAFE | OS/CPU/内存状态 |
| `bash.exec` | 执行命令 | NORMAL | 仅白名单命令 |
| `http.get` | 网页获取 | NORMAL | GET 请求，返回文本 |

**Bash 白名单**（轻量模式下可执行的命令）：
```
ls, cat, head, tail, grep, find, which, echo, pwd, date, whoami,
uname, df, du, ps, ifconfig, system_profiler(macOS), tasklist(Win)
```

### 7.4 助手模式额外工具

| ID | 名称 | 安全级别 | 说明 |
|----|------|:---:|------|
| `file.write` | 写文件 | DANGER | 写入/修改文件 |
| `file.delete` | 删除文件 | NOWAY | 默认禁止 |
| `bash.exec_full` | 全量命令 | DANGER | 覆盖白名单限制 |
| `app.open` | 打开应用 | NORMAL | 启动外部应用 |
| `clipboard.read` | 读剪贴板 | NORMAL | 读取剪贴板内容 |
| `clipboard.write` | 写剪贴板 | DANGER | 写入剪贴板 |
| `agent.spawn` | 子代理 | NORMAL | fork/team 模式 |

### 7.5 Skill 系统

Skill 写在 `skills/` 目录下，YAML frontmatter + Markdown 模板。助手模式启动时自动加载。

**Skill 文件格式**：

```yaml
# skills/summarize-code.md
---
id: summarize-code
name: 代码摘要
description: 分析指定代码文件/目录，生成结构化摘要
trigger_keywords: ["帮我看看代码", "分析这个项目", "这段代码做什么的"]
tools_needed: [file.read, file.list, file.search]
mode: assistant
safety: safe
---
# 执行步骤
1. 如果 path 是目录 → 调用 file.list 获取文件列表
2. 调用 file.read 读取关键文件
3. 分析代码结构，生成摘要：技术栈、主要模块、入口文件、关键依赖
4. 输出格式化摘要
```

**Skill 生命周期**：
```
Skill 文件(skills/*.md)
  → SkillLoader 解析 (助手模式启动时)
  → 提取 frontmatter + 执行模板
  → SkillRegistry 注册
  → 转换为 ToolDef 注入 ToolRegistry
  → 模型可直接调用
  → Skill 执行时 → 内循环调用 Local/MCP 工具 → 拼结果
```

### 7.6 MCP 集成（助手模式）

```
┌─────────────────────────────────────────────┐
│              MCP Manager                     │
│  ┌───────────────────────────────────────┐  │
│  │  MCPServerConfig[] (从 CONFIG 加载)    │  │
│  └───────────────────────────────────────┘  │
│                    │                         │
│  ┌─────────────────┴────────────────────┐   │
│  │          MCP Clients                  │   │
│  │  stdio: Rust spawn 子进程             │   │
│  │         TS←Rust←→stdin/stdout        │   │
│  │  SSE:   TS fetch() → HTTP SSE       │   │
│  │  每个 Client: connect → tools/list   │   │
│  │              → 转换ToolDef → 注入     │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

**MCP Server 配置**：
```yaml
tools:
  mcp:
    enabled: false           # 助手模式开关
    servers:
      - id: "filesystem"
        transport: "stdio"
        command: "npx"
        args: ["-y", "@anthropic/mcp-server-filesystem", "/path"]
        enabled: true
      - id: "remote"
        transport: "sse"
        url: "http://localhost:3001/sse"
        enabled: false
```

**MCP 生命周期**：
```
启动时:
  MCP Manager 读 CONFIG → tools.mcp.servers
  对每个 enabled server:
    stdio: invoke("mcp_spawn") → Rust spawn + 管道桥接
    sse:   fetch(url/sse) → EventSource
  连接后 → initialize → tools/list → 转ToolDef → 注入Registry

运行时:
  模型调用 MCP 工具 → ToolRouter → MCP Client → tools/call
  断开 → 自动重连 (指数退避, 最多3次)

关闭时:
  断开连接 → stdio 子进程 kill
```

## 九、安全控制系统

### 9.1 轻量模式安全

简洁策略：SAFE 直接放行，NORMAL/DANGER/NOWAY 直接拒绝（不弹确认）。

```
轻量模式 Safety:
  ToolCall → SafetyControl.check()
    SAFE → 放行
    NORMAL → 拒绝并人格化提示: "这个功能需要在助手模式下使用哦～"
    DANGER → 拒绝
    NOWAY → 拒绝
```

### 9.2 助手模式安全（四级+确认UI）

| 级别 | 标识 | 说明 | 示例 | 行为 |
|------|:---:|------|------|------|
| SAFE | 🟢 | 只读查询 | 读文件、系统信息 | 自动放行 |
| NORMAL | 🟡 | 需授权 | 打开应用、读剪贴板、HTTP GET | 首次确认后可会话内信任 |
| DANGER | 🟠 | 需确认 | 写文件、全量命令、写剪贴板 | 每次都确认 |
| NOWAY | 🔴 | 硬禁止 | sudo、格式化、rm -rf、SSH 密钥 | 直接拒绝 |

## 十二、关键调用链

### 12.1 轻量模式：日常查询

```
用户: "桌面有什么文件"
  → PreProcessor → 非命令
  → 人格中间件.wrap("thinking") → expression: "thinking"
  → 思考强度: auto → 检测"什么/文件" → medium
  → ContextEngine.build()
     ├── 人格 Prompt
     ├── 轻量工具声明 (6个, ~150 tokens)
     └── Memory 注入
  → AI → tool_call: file.list({ path: "~/Desktop" })
  → 人格中间件.wrap("executing") → "让我看看桌面有什么..."
  → Safety: SAFE → 放行
  → 执行: Rust read_dir → 文件列表
  → 结果回注 → AI 生成自然语言
  → "桌面上有3个文件夹和2个文件呢～Documents、Pictures、Downloads..."
  → 人格中间件.wrap("done") → "happy", sound: "reply"
  → ChatPanel 显示 + StreamView 表情
```

### 12.2 助手模式：复杂工具链

```
用户: "帮我整理桌面，把截图放到一个文件夹"
  → PreProcessor
  → 思考强度: auto → 检测"整理/文件夹" → high
  → ContextEngine: 助手模式全量工具声明
  → Plan: 1.列文件 2.找截图 3.创建文件夹 4.移动
  → [Round 1] AI → file.list("~/Desktop")
     → SAFE → 放行 → 结果回注
  → [Round 2] AI → bash.exec("find ~/Desktop -name '*.png' -o -name '*.jpg'")
     → DANGER → 确认UI → 用户确认 → 执行 → 结果回注
  → [Round 3] AI → bash.exec("mkdir -p ~/Desktop/截图整理")
     → DANGER → 确认 → 执行
  → [Round 4] AI → bash.exec("mv ~/Desktop/*.png ~/Desktop/*.jpg ~/Desktop/截图整理/")
     → DANGER → 确认 → 执行
  → 最终回复: "搞定啦～已经把截图整理到'截图整理'文件夹了♡"
```

### 12.3 MCP 工具调用（助手模式）

```
用户: "数据库里有什么表"
  → MCP Server "sqlite" 已连接，tools 已注入
  → 上下文包含 MCP 工具声明
  → AI → tool_call: mcp.sqlite.list_tables({})
     → Safety: NORMAL → 首次确认
     → MCP Client → tools/call JSON-RPC → sqlite server
     → 返回表列表
  → AI → "这个数据库有3个表：users、orders、products..."
```

## 十八、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 双模式 | 同一Loop+不同工具集 | 不搞两套代码，零重复 |
| 轻量能力 | 6个基础工具 | 覆盖日常查询，不残废 |
| 思考强度 | auto 自动调节 | 闲聊快响应，复杂任务深推理 |
| 人格定位 | 横切中间件 | 所有阶段都要角色化 |
| MCP | 助手模式 | 普通用户不需要外部工具 |
| Skill | 助手模式 | 编排式工作流属高级功能 |
| 工具声明(轻量) | 始终携带 | 只有6个，不占token，省决策 |
| 工具声明(助手) | 按需三级 | 工具多，必须分级省token |
| Rust 职责 | 保持轻薄 | OS调用+MCP桥接，逻辑在TS |
| 状态管理 | Vue reactive | UI响应式 |
| 记忆存储 | 文件MD | 可读可迁移 |
